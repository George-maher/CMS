1. PROJECT OVERVIEW
================================================================================

The Church Management System (CMS) is a production-grade platform designed to
manage church operations including member management, attendance tracking via QR
codes, points/leaderboard gamification, event management, feedback systems, and
role-based administration. It serves multiple churches through a platform admin
model where each church operates as an independent tenant with its own admin,
servants, and members.

The system is built with a clean architecture pattern in Laravel 12 backend,
React 19 with TypeScript frontend, PostgreSQL database, and Docker containerization.
It uses Laravel Sanctum for API authentication and Resend for email delivery.

Key Capabilities:
- Multi-tenant church management
- QR code-based attendance system
- Invite-based registration (admin-to-servant, servant-to-member)
- Points and leaderboard gamification for member engagement
- Event management with viewing analytics
- Feedback system with anonymous submissions
- Daily verse management
- Attendance context system (Sunday School, Mass, Trips, etc.)
- Role-based dashboards (Platform Admin, Church Admin, Assistant Admin, Servant, Member)
- Bilingual support (English + Arabic)
- Dark mode / Light mode theming

4. BACKEND ARCHITECTURE
================================================================================

4.1 Architectural Pattern
-------------------------
The backend follows a Service-Repository pattern with dependency injection:

Controller → Service (Business Logic) → Repository (Data Access) → Model (Eloquent)

- Controllers handle HTTP requests/responses using the ApiResponseTrait
- Services contain all business logic (no business logic in controllers)
- Repositories encapsulate database queries
- Models define relationships and scopes
- Contracts (interfaces) enable loose coupling and testability
- DTOs (Data Transfer Objects) for structured data passing
- Form Requests handle validation
- API Resources transform model data for JSON responses
- Policies handle authorization
- Enums provide type-safe constants
- Traits provide reusable functionality across models/classes

4.2 Service Layer (25 services)
--------------------------------
Services are bound to interfaces in AppServiceProvider::register().

Core Services:
- AuthService          - Login, register, logout, password reset, email verification
- AttendanceService    - Record attendance, history, stats, absent members, context analytics
- UserService          - User CRUD, promotion, demotion, member management
- QRInviteService      - Generate QR invites, validate tokens, accept invites
- PointService         - Add points, balance, history, leaderboard, bonus points
- EventService         - Event CRUD, filtering, viewing
- FeedbackService      - Submit feedback, reply, resolve
- StageService         - Stage CRUD, bulk operations
- ClasseService        - Class CRUD, servant/member assignments, reordering
- LeaderboardService   - Global, class-specific, and stage-specific leaderboards
- MembershipRequestService - Submit, approve, reject membership requests
- PasswordResetRequestService - Admin-approved password reset flow
- AttendanceContextService - Context management (Sunday School, Mass, etc.)
- VerseService         - Daily verse CRUD and activation
- CacheService         - Singleton caching helper
- NotificationService  - In-app notification management
- EmailService         - Email sending via Resend
- AuditService         - Activity logging
- SupabaseStorageService - File storage on Supabase
- FileUploadService    - File upload handling
- StorageService       - Abstraction for storage operations
- ChurchApplicationService - Church registration application handling
- ChurchDeletionService - Church soft/hard delete with recovery
- EventAnalyticsService - Event view tracking and analytics

5. FRONTEND ARCHITECTURE
================================================================================

5.1 Component Architecture
--------------------------
The frontend uses a component-based architecture with lazy loading:

- Each page is a lazy-loaded component (React.lazy + Suspense)
- Shared components are organized by domain (common/, layout/, leaderboard/)
- API layer is separated into per-resource modules
- Contexts provide global state (Auth, Theme)
- Custom hooks encapsulate reusable logic

5.2 State Management
--------------------
- AuthContext - Manages user authentication, token, and user data
- ThemeContext - Manages theme (light/dark) and language (en/ar)
- No Redux or Zustand - state is managed via Context + useState/useReducer

5.3 API Layer
-------------
- Centralized Axios instance (client.ts)
- Request interceptor adds Bearer token from localStorage
- Response interceptor handles 401 (auto-logout) and 429 (auto-retry with backoff)
- Per-resource API modules export typed functions
- Base URL: /api/v1 (proxied through Vite dev server)

5.4 Routing
-----------
- react-router-dom v7 with nested routes
- AppLayout component wraps authenticated routes with role-based access
- Public routes: Landing, Join, Login, ForgotPassword, Invite, etc.
- Authenticated routes grouped by role:
  - /platform/* - Platform Admin
  - /admin/* - Church Admin
  - /assistant-admin/* - Assistant Admin
  - /servant/* - Servant
  - /member/* - Member

5.5 Theme System
----------------
- TailwindCSS v4 with CSS custom properties
- Light/Dark mode via :root.dark class toggle
- Gold/Navy color palette with full dark mode inversion
- Glassmorphism design with backdrop-blur
- Custom utility classes: .btn-gold, .btn-primary, .card, .glass-card, etc.

5.6 Internationalization
------------------------
- i18next + react-i18next
- Bilingual: English (en.json) and Arabic (ar.json)
- RTL support via dir attribute on <html>
- Language detection from localStorage
- Arabic font: Cairo/Tajawal in RTL mode

5.7 Key UI Components
--------------------
- DataTable - Responsive table with card view on mobile, pagination
- Modal - Portal-based modal with sizes (sm/md/lg/xl/full)
- StatCard - Dashboard statistics card with icon and color variants
- Badge - Status badge (success, warning, danger, info, gold)
- LoadingSpinner - Full-page or inline loading indicator
- ImageUpload - File upload with preview
- ImageWithFallback - Image with fallback on error
- CopyButton - One-click copy to clipboard
- EventDetailModal - Detail modal for events
- DailyVerseBanner - Verse of the day banner
- MotionDiv - Animated wrapper with staggered entry
- PodiumCard / LeaderboardRow - Leaderboard display components

- Foreign keys with CASCADE on most child tables
- Composite unique indexes on attendances preventing duplicates
- Performance indexes on created_at, church_id, role, class_id, etc.
- Scalability indexes on frequently filtered/joined columns

7. AUTHENTICATION & AUTHORIZATION
================================================================================

7.1 Authentication
-------------------
- Laravel Sanctum token-based API authentication
- Login returns a Bearer token stored in localStorage
- Token has 24-hour expiration (configurable in config/sanctum.php)
- Token payload includes user role as abilities
- SPA-style stateful domains configured in SANCTUM_STATEFUL_DOMAINS
- Platform Admin login uses a hidden path (platform admin login route)
- Email verification via token sent on registration
- Password reset via standard Laravel broker OR admin-approved flow

7.2 Authorization System
-------------------------
Two-layer authorization:

Layer 1: Role-based (via middleware `role:`)
- Platform Admin: Full system access, manages all churches
- Admin: Church-level administration
- Assistant Admin: Same as Admin (slightly less permission)
- Servant: Manages members, records attendance, creates invites
- Member: Views own data, submits feedback

Layer 2: Permission-based (via PermissionMiddleware)
- Permissions are tied to roles via the Permission model
- Permissions include:
  manage_users, view_users, manage_invites, record_attendance,
  manage_events, manage_verses, manage_feedback, submit_feedback,
  manage_membership_requests, view_points, manage_attendance_contexts
- Each permission maps to one or more roles statically defined
- HasPermissions trait provides user-level permission checking

7.3 Approved Middleware
-----------------------
- Ensures the authenticated user has application_status = 'approved'
- Blocks pending/rejected users from accessing protected routes

7.4 Rate Limiting
-----------------
28 named rate limiters covering:
- API general: 300 req/min (authenticated), 60 req/min (guest)
- Login: 5 attempts/min per IP/email
- Register: 10 registrations/hour per IP
- Sensitive operations: 10 req/min
- File uploads: 10/min, imports: 5/hour
- Invites: 20/hour generation, 10/hour validation
- Attendance: 100/min record, 60/min read
- All return standardized 429 responses with Retry-After header

9. FLOW ANALYSIS
================================================================================

9.1 User Registration Flow (via Invite)
----------------------------------------
1. Admin/Servant generates QR invite with specific role
2. New user receives invite URL (via QR or link)
3. User opens InviteLanding → validates token
4. If valid & not expired & not used → shows registration form
5. If authenticated (existing user) → shows accept invite button
6. User fills registration form (name, email, password, class, etc.)
7. AuthService::register() called
8. Validates invite with lockForUpdate() (race condition prevention)
9. Creates user with hashed password
10. Generates email verification token
11. Sends verification email
12. Marks invite as used
13. User redirected to login
14. User must verify email before full access

9.2 Login Flow
---------------
1. User submits email + password
2. AuthService::login() checks credentials
3. Blocks: platform_admin (must use special login), inactive users,
           suspended church, deleted church
4. Creates Sanctum token with role ability
5. Returns user data + token
6. Frontend: stores token in localStorage, stores user in localStorage
7. AuthContext validates token by calling /auth/me
8. User redirected to role-appropriate dashboard

9.3 Attendance Recording Flow
------------------------------
1. Servant opens ScanQR page
2. Selects attendance context (mandatory) + optional event + optional class filter
3. Two methods:
   a. QR Scan: Opens camera → scans member's QR token → handles decoded text
   b. Manual: Enters QR token or member ID
4. System calls lookupByToken/lookupByMemberId → validates
5. Confirmation dialog shown with member info
6. Servant confirms → AttendanceService::recordAttendance() called
7. Validates: member active, context active, event exists
8. lockForUpdate() on member row
9. Checks hasAttendanceToday() → prevents duplicate same context
10. Creates attendance record
11. Auto-awards 10 points via PointService
12. Dispatches AttendanceRecorded event
13. Returns attendance + points earned

9.4 Forgot Password Flow (Standard)
-----------------------------------
1. User submits email on ForgotPassword page
2. AuthService::forgotPassword() calls Laravel Password broker
3. Email sent with reset link (frontend URL)
4. User clicks link → ResetPassword page
5. Submits new password with token
6. AuthService::resetPassword() validates and updates
7. Invalidates all existing Sanctum tokens
8. Sends PasswordChangedNotification

9.5 Forgot Password Flow (Admin-Approved)
------------------------------------------
1. User submits email + optional notes on ForgotPassword page
2. PasswordResetRequestService::submitRequest() creates pending request
3. Admin receives email notification
4. Admin reviews request detail (user info, phone, class, role)
5. Admin contacts user externally (phone/WhatsApp) to verify identity
6. Admin approves or rejects with reason
7. On approval: email sent to user with secure reset URL (24-hour token)
8. User clicks link → ResetPasswordFromRequest page
9. User sets new password
10. Token marked used, Sanctum tokens invalidated

9.6 Event Creation Flow
------------------------
1. Servant/Admin opens Events page
2. Clicks "Create Event" button
3. Modal opens with form: name, type, description, date, location, image, target
4. Target can be: all classes or specific classes
5. Image uploaded via FileUploadService to Supabase
6. Event created with creator tracking
7. Event visible to targeted members

9.7 Event Analytics Flow
-------------------------
1. Member views event → EventController::show() + trackView()
2. trackView() creates/updates EventView record
3. Admin/Servant views analytics:
   - Summary: view count, target member count, view rate
   - Viewed: list of members who viewed
   - Not viewed: list of members who haven't viewed

9.8 Feedback Flow
------------------
1. Member opens FeedbackSubmit page
2. Selects category (complaint/suggestion/other) and optionally anonymous
3. Submits message
4. Servants/admins see feedback in FeedbackManagement
5. Admin sees full sender identity; servant sees anonymous if flagged
6. Admin/servant can reply
7. Admin can mark as resolved/unresolved
8. Member notified of new reply

9.9 Church Application Flow
----------------------------
1. Church leader fills JoinNow form (church details, admin credentials, ID docs)
2. Form submitted with front_id, back_id (or church permission document)
3. Platform admin reviews in PlatformDashboard
4. Admin checks docs, can approve/reject with reason
5. On approval:
   - ChurchApplicationObserver fires
   - Creates Church record with default attendance contexts
   - Creates admin user for the church
   - Sends approval notification

9.10 Points & Leaderboard Flow
-------------------------------
1. Attendance recorded → auto-award 10 points (type: attendance)
2. Admin can award bonus points (type: bonus)
3. LeaderboardService calculates rankings:
   - Global: all members across all classes
   - By class: members within a specific class
   - By stage: members within a stage (across classes)
   - Servant's classes: all classes assigned to servant
4. Podium display: top 3 with gold/silver/bronze styling
5. Points are summed; ties possible

16. PRESENTATION SUMMARY
================================================================================

Project Title: Church Management System (CMS)

────────────────────────────────────────────────────────────────────────────
16.1 Project Overview
────────────────────────────────────────────────────────────────────────────

The Church Management System is a comprehensive digital platform designed to
streamline church operations, enhance member engagement, and provide powerful
administrative tools for church leaders. It serves as a complete church
management solution with QR-based attendance tracking, gamification through
points and leaderboards, event management, feedback systems, and role-based
access control.

The system supports multiple churches on a single platform, with each church
operating independently with its own administrators, servants, and members.
This makes it ideal for denominations or church networks that want a unified
system while maintaining church-level autonomy.

────────────────────────────────────────────────────────────────────────────
16.2 Problem Statement
────────────────────────────────────────────────────────────────────────────

Churches face several operational challenges:
- Manual attendance tracking is time-consuming and error-prone
- Member engagement is difficult to measure and encourage
- Communication between servants and members is fragmented
- Event planning lacks analytics to measure reach
- Feedback collection and management is ad-hoc
- No centralized system for managing multiple church groups

Traditional solutions are either too generic (not church-specific), too
expensive, or lack modern features like QR scanning, mobile responsiveness,
and bilingual support.

────────────────────────────────────────────────────────────────────────────
16.3 Solution
────────────────────────────────────────────────────────────────────────────

The Church Management System provides:

- QR-based attendance: Servants scan members' QR codes for instant check-in
- Gamification: Points and leaderboards encourage regular attendance
- Role-based access: Platform Admin → Church Admin → Servant → Member
- Multi-tenant architecture: One platform, many churches
- Event management: Create, target, track, and analyze events
- Feedback system: Anonymous submissions with reply capability
- Daily verses: Bible verse display for spiritual engagement
- Bilingual: Full English and Arabic support
- Modern UI: Responsive design with dark mode

────────────────────────────────────────────────────────────────────────────
16.4 System Architecture
────────────────────────────────────────────────────────────────────────────

The system follows a modern three-tier architecture:

┌──────────────────────────────────────────────────────────────┐
│                   Client Layer (React + Vite)                │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────┐    │
│  │ Admin UI │ │ Servant  │ │ Member   │ │ Platform     │    │
│  │          │ │    UI    │ │    UI    │ │ Admin UI     │    │
│  └──────────┘ └──────────┘ └──────────┘ └──────────────┘    │
└──────────────────────────┬───────────────────────────────────┘
                           │ HTTP/JSON (REST API)
┌──────────────────────────┴───────────────────────────────────┐
│                   API Layer (Laravel 12)                     │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────┐    │

16.6 Main Features
────────────────────────────────────────────────────────────────────────────

1. Multi-Tenant Church Management
   - Platform admin oversees multiple churches
   - Each church operates independently
   - Church-level data isolation

2. QR-Based Attendance System
   - Secure 64-char token QR codes
   - Camera scanning or manual token entry
   - Duplicate prevention (same day + context)
   - Context-aware (Sunday School, Mass, Trip, etc.)
   - Automatic point awarding

3. Points & Leaderboard Gamification
   - Auto-points on attendance (10 points)
   - Bonus points by admins
   - Global, class, and stage leaderboards
   - Podium display for top 3

4. Role-Based Dashboards
   - Platform Admin: Global oversight, application management
   - Church Admin: Full church management
   - Assistant Admin: Co-management
   - Servant: Member management, attendance recording
   - Member: Personal dashboard, QR code, leaderboard

5. Event Management with Analytics
   - Create events with images and targets
   - Track member views
   - View/not-viewed analytics
   - Targeted to specific classes or all

6. Feedback System
   - Anonymous submission option
   - Category-based (complaint/suggestion/other)
   - Reply and resolution workflow
   - Admin override on anonymity

7. Bilingual Support
   - Full English and Arabic interface
   - RTL layout support
   - Bilingual email notifications

8. Secure Invite System
   - Admin-to-servant and servant-to-member invites
   - Configurable expiration
   - Single or multi-use
   - Revokable

────────────────────────────────────────────────────────────────────────────
16.7 System Workflow
────────────────────────────────────────────────────────────────────────────

Church Onboarding:
  Church Leader → Submits Application → Platform Admin Reviews →
  Approves/Rejects → Church Created with Default Contexts →
  Church Admin Account Created → Church Active

Member Onboarding:
  Admin/Servant → Generates QR Invite → Shares with New User →
  User Opens Invite Link → Registers with Details →
  Email Verification → Login → Dashboard Access

Daily Attendance:
  Servant Opens Scan QR → Selects Context →
  Scans Member QR or Enters Member ID →
  System Validates → Records Attendance →
  Awards Points → Updates Leaderboard

Member Experience:
  Login → Dashboard (Points, QR Code, Recent Attendance, Upcoming Events) →
  View Attendance History → Check Leaderboard →
  Submit Feedback → View Events → Scan Own QR

────────────────────────────────────────────────────────────────────────────
16.8 Database Overview
────────────────────────────────────────────────────────────────────────────

25+ database tables with the following core entities:

- Users: 5 roles, church-scoped, self-referential (servant-member relationship)
- Churches: Multi-tenant hierarchy, soft-deletable with recovery
- Attendances: Context-aware, duplicate-prevented, auto-points
- Points: Type-based (attendance/bonus/adjustment), church-scoped
- QR Invites: Type-based with expiration, usage tracking, optimistic locking
- Events: Targetable to classes, with view tracking analytics
- Feedback: Anonymous option, reply workflow, resolved tracking
- Classes/Stages: Educational hierarchy, servant assignments via pivot
- Notifications: In-app system with read tracking

PostgreSQL-specific features used: sequences (for member_id auto-generation),
unique indexes, foreign key constraints with cascade.

────────────────────────────────────────────────────────────────────────────
16.9 Security
────────────────────────────────────────────────────────────────────────────

- Token-based authentication (Sanctum) with 24-hour expiry
- Role-based and permission-based authorization
- Rate limiting on all endpoints (28 limiters)
- QR tokens are secure random strings (never raw IDs)
- Race condition prevention (lockForUpdate, optimistic locking)
- Hidden platform admin login path
- Password hashing (bcrypt)
- Email verification requirement
- Church-level data isolation
- Soft deletes for recovery
- Audit logging

Email via Resend API. File storage via Supabase. Database via PostgreSQL.
